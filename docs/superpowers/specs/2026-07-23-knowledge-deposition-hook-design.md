# Knowledge Deposition Hook Design

**Date:** 2026-07-23

**Status:** Approved

## Goal

Add a task-end knowledge deposition gate for Codex and Claude Code. Every Stratum task must produce an auditable report
that either lists knowledge candidates or explicitly records that no durable knowledge was produced. Reports are local,
ignored artifacts under `tmp/`; the gate never writes Skills, hooks, global instructions, project documentation, or the
Obsidian vault automatically.

## Success Criteria

- Codex and Claude Code use the same classification policy and report schema.
- Every completed Stratum task turn has a fresh report associated with its client session and task ID, including
  `decision: none` reports; a report from an earlier task cannot satisfy a later turn.
- A missing, invalid, or cross-session report prevents the client from stopping and gives the Agent an actionable error.
- Concurrent sessions do not overwrite or satisfy each other's gate.
- Reports contain evidence references and boundaries without storing transcripts, prompts, secrets, or raw upstream data.
- Existing generated `AGENTS.md` and `CLAUDE.md` remain derived from `docs/agent/instructions.md` and their prefixes.
- Non-Stratum repositories are unaffected by the user-level hook registrations.

## Existing Policy And Ownership

The existing knowledge protocol already requires end-of-task classification and a `Knowledge deposition candidates`
summary. This feature mechanizes that policy instead of introducing a second policy.

Ownership is split as follows:

| Component | Responsibility |
|---|---|
| `docs/agent/knowledge-deposition.md` | Canonical Stratum classification policy and report contract |
| `docs/agent/instructions.md` | Short mandatory rule and link to the canonical policy |
| Agent | Semantic classification, evidence collection, Obsidian duplicate search, and candidate content |
| Project report command | Schema validation, safe persistence, Markdown rendering, and latest pointer update |
| Client hooks | Session binding, task-end enforcement, and actionable feedback |

`AGENTS.md` and `CLAUDE.md` are regenerated by the existing generator. They must not gain independently maintained
copies of the detailed classification policy.

## Classification Policy

One task can contain multiple candidates and one claim can target multiple destinations only when each destination has a
distinct consumption purpose.

| Destination | Inclusion rule | Exclusions |
|---|---|---|
| `skill` | Repeatable Agent procedure, checklist, tool method, or operational pitfall that should load on demand | Single rules, current project state, incidental commands |
| `hook` | Rule that can be detected, blocked, prompted, or repaired deterministically | Semantic judgments with broad legitimate exceptions |
| `global_md` | Cross-project principle, stable preference, or safety boundary that should remain in default context | Project-specific rules, detailed tutorials, infrequent knowledge |
| `obsidian` | Cross-project durable fact, principle, case, counterexample, or correction with evidence and applicability boundaries | Transient state, raw logs, full conversations, search snippets, secrets, unsupported guesses |
| `project_git` | Stratum-specific architecture, interface, ADR, test contract, or verified runtime fact | Generic knowledge whose value is not tied to Stratum |
| `none` | Routine work, duplicate knowledge, temporary state, or insufficiently supported observations | Used only when the task has no valid candidate |

An `obsidian` candidate also records its knowledge type (`fact`, `principle`, `case`, `counterexample`, or
`correction`), Vault queries used for duplicate checking, the closest existing notes, verification status, version or
scope boundaries, and a recommended governance action (`create`, `merge`, `correct`, or `queue`). The normal coding
task remains read-only with respect to Obsidian.

When a principle and an executable procedure derive from the same evidence, the principle can target `obsidian` while
the procedure targets `skill`. Each candidate describes its own consumption purpose and links to the other instead of
copying the same body.

## Report Contract

The JSON report is the machine-readable source of truth. The report command renders Markdown from JSON; Agents do not
maintain the two formats independently.

Required top-level fields:

- schema version;
- client (`codex` or `claude`);
- sanitized session ID;
- hook-generated task ID;
- repository identity and worktree root;
- UTC creation time;
- decision (`candidates` or `none`);
- task summary;
- candidate array;
- `none` reason when the decision is `none`.

Every candidate contains:

- stable candidate ID;
- one-sentence claim;
- destination;
- evidence paths and optional line anchors;
- applicability scope and explicit exclusions or counterexamples;
- duplicate-check result;
- recommended target;
- confidence;
- consumption purpose when the same claim has multiple destinations.

The schema rejects unknown destinations, missing evidence, absolute evidence paths, parent traversal, embedded secret
fields, and inconsistent combinations such as `decision: none` with candidates. Evidence may reference repository files,
test names, command identifiers, or governed Obsidian note paths, but the report does not copy raw evidence bodies.

Reports are stored as:

```text
tmp/knowledge-deposition/
├── YYYY-MM-DD/
│   ├── codex-<session-id>-<task-id>.json
│   ├── codex-<session-id>-<task-id>.md
│   ├── claude-<session-id>-<task-id>.json
│   └── claude-<session-id>-<task-id>.md
├── current/
│   ├── codex-<session-id>.json
│   └── claude-<session-id>.json
└── latest.md
```

Every JSON/Markdown pair includes the task ID generated by the prompt hook. `latest.md` is an atomically replaced
link/index for convenience, not a second history store. The dated report files are authoritative local artifacts. A
client/session current-task marker stores the active task ID, so a later prompt cannot reuse an earlier report. The
existing `tmp/` ignore rule prevents report churn from entering Git.

## Session And Hook Flow

```text
UserPromptSubmit
  -> verify this is the Stratum repository
  -> sanitize client session ID
  -> generate and atomically store a unique task ID
  -> inject the session/task report identity and report command into Agent context

Agent finishes substantive work
  -> inspect project evidence and applicable knowledge sources
  -> classify candidates or choose none
  -> invoke the project report command with the bound client/session/task identity
  -> receive the generated report paths

Stop
  -> verify this is the Stratum repository
  -> read the current client/session task marker
  -> locate the exact client/session/task JSON report
  -> validate schema, repository identity, and task freshness
  -> allow stop when valid
  -> otherwise block stop and return the precise remediation command/error

Final response
  -> summarize candidates and link the Markdown report
```

The Claude and Codex adapters remain separate because their hook input and output protocols differ. They may invoke a
shared project-owned validation/reporting core, but neither adapter imports or forwards to the other client's hook
directory.

The Stop adapters must honor each client's re-entry indicator. A second Stop evaluation is allowed after the Agent
creates or repairs a report; an invalid report never becomes valid merely because the hook is already active.

## Main-Branch And Concurrency Behavior

Read-only tasks often run in the primary checkout, so the exact report command must be allowed to write only beneath
the ignored `tmp/knowledge-deposition/` directory. This is a narrow exception in the worktree policy, not a general Bash
or main-branch write exemption. The report command resolves and validates its destination internally and rejects caller-
supplied output paths.

The prompt adapter uses a repository-local lock to atomically replace the current task marker. The writer uses the same
lock, writes each JSON and Markdown artifact to a temporary file in the destination directory, validates it, and renames
it atomically. The `latest.md` update uses the same lock. Session/task-specific names prevent concurrent sessions and
successive tasks in one session from overwriting or satisfying one another. Stop is read-only and cannot consume a report
before other Stop hooks have completed.

## Failure And Security Behavior

- Malformed hook input, missing dependencies, invalid repository identity, write failure, schema failure, and missing
  reports fail visibly. The Stop hook blocks with an actionable reason for Stratum sessions.
- Non-Stratum repositories exit successfully without creating files or injecting project instructions.
- Session IDs are allowlisted to safe filename characters and length; raw IDs never become arbitrary paths.
- Reports never include transcript text, full prompts, environment values, credentials, bearer tokens, raw API responses,
  or customer-sensitive content.
- Obsidian remains read-only. A report recommends a governance action but does not call a Vault write tool.
- Report retention and cleanup are not part of request or startup paths. Any later cleanup requires a separately reviewed,
  explicit maintenance command.

## Configuration

Project-owned scripts and policy are committed to Git. User-level Codex and Claude hook registrations call the respective
project adapter only when the hook input resolves to Stratum. Registration changes are tested and documented, while
machine-specific notification and unrelated hooks remain untouched.

The detailed policy is indexed from `docs/agent/instructions.md`. Regeneration updates both root instruction files, and
the existing drift check guards their generated status.

## Verification

Focused tests cover:

- all six destinations and a multi-destination task;
- a valid `none` report;
- missing fields, unknown categories, inconsistent decisions, absolute/traversing paths, and forbidden sensitive fields;
- deterministic Markdown rendering from JSON;
- valid, missing, malformed, stale, and cross-session reports for both clients;
- rejection of an earlier task's report after a new prompt changes the current task marker;
- Stop hook re-entry after remediation;
- concurrent writers and atomic `latest.md` updates;
- the narrow main-checkout report-command allowance and denial of unrelated writes;
- clean no-op behavior outside Stratum;
- syntax and JSON parsing for both user-level hook configurations;
- existing `.claude/hooks/run-tests.sh`, `.codex/hooks/run-tests.sh`, instruction generation checks, and risk guardrails.

Because this feature changes Agent task-finalization behavior, end-to-end verification must exercise real Codex and Claude
hook payloads or the closest supported local harness, confirm files are created under `tmp/`, verify a missing report is
blocked, and verify a corrected report permits completion. Any client limitation that prevents a true Stop event must be
reported rather than represented as covered by a synthetic test.

## Out Of Scope

- Automatically writing, merging, or deleting Skills, hooks, global instructions, project documentation, or Vault notes.
- An independent LLM call from a Stop hook.
- Heuristic classification based only on Git diff or command history.
- A cross-project knowledge governance queue or automatic report retention policy.
- Copying task transcripts into reports for later classification.
