# Agent 面试资料分类去重与增量融合设计

## 目标

将 `tmp/agent-interview/reports/` 下按日期生成的 Agent 高级岗位研究报告，重组为按 Agent 工程领域维护的长期资料库。首轮融合成功后删除原始日期报告；后续定时任务先生成临时报告，再按索引规则分类、去重并融合到既有文件。

系统必须保留报告中的来源、趋势、面试题、Stratum 可补强点和跟踪关键词。任何无法稳定分类的内容进入固定的待分类文件，不允许无人值守任务自行创建分类。

## 非目标

- 不把 `tmp/` 下的本地研究资料改为受版本控制的项目事实源。
- 不把市场趋势或自动生成建议视为已经验证的仓库事实。
- 不自动修改 Stratum 业务代码来落实报告中的改进建议。
- 不在本次工作中迁移 Hermes 调度框架或其他 `tmp/cron/` 任务。
- 不保留首轮融合后的日期原报告或旧的日报 `latest.md` 链接。

## 固定分类

`tmp/agent-interview/reports/` 只允许包含以下长期文件：

| 文件 | 收录范围 |
|---|---|
| `01-agent-runtime-and-workflow.md` | Agent loop、ReAct、Plan-and-Execute、Workflow、DAG、checkpoint、恢复和多 Agent 协作 |
| `02-tools-mcp-and-approval.md` | Tool calling、Skill、MCP、工具 schema、副作用、审批和 Human-in-the-loop |
| `03-context-and-memory.md` | 上下文预算、历史压缩、长期记忆、事实生命周期和 Memory Pipeline 语义 |
| `04-knowledge-and-rag.md` | Knowledge、RAG、检索评测、摄取、embedding 与向量迁移 |
| `05-llm-gateway-and-model-routing.md` | Provider 抽象、模型路由、fallback、流式协议和模型成本策略 |
| `06-reliability-and-streaming.md` | timeout、retry、circuit breaker、幂等、背压、消息可靠性、SSE 和资源生命周期 |
| `07-evaluation-observability-and-cost.md` | Agent eval、trace、SLO、发布门禁、token、成本和容量证据 |
| `08-security-iam-and-multitenancy.md` | Prompt Injection、IAM、OAuth/JWT、凭据、租户隔离和审计 |
| `09-architecture-and-production-readiness.md` | DDD 边界、平台化、Harness、生产就绪标准和 Staff 级技术治理 |
| `99-unclassified.md` | 无法依照固定规则唯一分类的内容 |
| `README.md` | 人工导航、机器分类契约、处理账本和融合状态 |

跨分类内容只能选择一个主分类。`README.md` 为相邻分类定义判定优先级和排除规则；其他分类可通过稳定条目 ID 链接主条目，不复制正文。无人值守任务不得创建上述列表之外的分类文件。

## 分类文件模型

每个领域文件使用一致结构：

```markdown
# 分类名称

## 分类边界

## 趋势与观点

## 面试题

### <稳定 ID> <规范标题>
- 作答要点
- 深挖问题
- Stratum 实现与边界
- 相关源码/文档
- 来源
- 首次收录 / 最近更新

## Stratum 可补强点

## 跟踪关键词

## 参考来源
```

趋势、问题和补强点均使用稳定 ID。稳定 ID 来源于内容类型、主分类和规范主题，而不是日报文件名或原始序号。标题措辞变化但主题相同的内容必须融合到同一条目。

`Stratum 实现与边界` 必须区分代码或文档可以确认的现状、报告提出的建议以及尚未复核的历史判断。新报告不得仅凭较新日期覆盖与当前仓库证据冲突的内容；冲突需保留适用范围或标记为待复核。

参考来源统一记录标题、URL、来源类型、首次收录日期和最近核验日期。领域条目引用来源 ID，避免在多个问答中重复完整链接说明。

## 索引与处理账本

`README.md` 同时供人和任务读取，包含：

1. 分类文件导航和每类收录边界；
2. 分类关键词、排除规则和跨分类优先级；
3. 稳定 ID、规范化和去重规则；
4. 已处理输入的运行 ID、报告日期、SHA-256 摘要和融合统计；
5. 最近一次成功融合时间；
6. 原始条目数、融合后条目数、重复条目数和待分类条目数；
7. `99-unclassified.md` 的人工处理说明。

输入账本以内容摘要而不是文件名作为幂等依据。同一内容换名后再次输入仍应被识别为已经处理。账本不保存整份原始报告。

## 去重与融合规则

去重按从确定到语义的顺序执行：

1. 对标题做大小写、空白、标点和常见中英文术语归一化；
2. 比较稳定主题、内容类型、主分类、源码路径和来源；
3. 对候选条目执行语义判断，确认是重复、补充、冲突还是新主题；
4. 重复内容不写入；补充内容合并到既有字段；冲突内容保留边界或待复核标记；新主题创建稳定条目；
5. 无法唯一分类或无法可靠判断的内容写入 `99-unclassified.md`。

融合时优先保留信息完整、仓库证据明确、来源可核验的表达。日期较新只能影响“最近更新”和来源核验信息，不能单独作为覆盖旧内容的理由。跟踪关键词使用归一化后的集合去重；来源按规范 URL 去重。

## 定时任务数据流

现有 `tmp/cron/daily-agent-interview.sh` 调整为两个阶段：

1. **生成阶段**：研究结果写入 `tmp/agent-interview/reports/inbox/<run-id>.md`，不直接修改长期分类文件。
2. **融合阶段**：读取 `README.md` 和全部固定分类文件，在同一文件系统内创建临时工作目录，完成分类、去重、融合和索引更新。
3. **验证阶段**：验证分类白名单、Markdown 结构、稳定 ID、来源、输入账本、待分类数量和输入覆盖率。
4. **发布阶段**：验证全部通过后原子替换正式分类文件，删除已融合 inbox 输入，并让 `latest.md` 指向 `README.md`。

Codex 负责语言归并和语义分类；脚本硬编码允许的文件集合、阶段顺序、哈希、校验、发布、删除和失败码。模型不得决定是否绕过校验、是否删除输入或是否把失败当成成功。

任务持有现有专用文件锁，防止两个运行同时融合。输入报告在发布成功前始终保留。若融合超时、Codex 失败、输出不合法或发布失败，正式资料库保持原样，任务返回非零；下一次运行可重试 inbox 中的输入。

## 首轮历史融合

首轮将当前 14 份日期报告作为一个批次：

1. 计算每份报告摘要并记录原始问题、趋势、补强点、关键词和来源数量；
2. 创建固定分类文件及 `README.md`；
3. 拆分并融合全部内容，将不能稳定分类的条目写入 `99-unclassified.md`；
4. 验证每份报告均进入处理账本；
5. 验证每个原始问题至少映射到一个稳定条目或待分类条目；
6. 输出首轮融合统计；
7. 仅在上述检查全部通过后删除 14 份日期报告及旧 `latest.md`，再创建指向 `README.md` 的新 `latest.md`。

删除后的原始报告不另建 archive。处理账本、条目首次收录日期和来源记录承担必要的来源追溯，但不保证逐字恢复日报。

## 校验与测试

融合工具提供独立检查模式，并覆盖以下行为：

- 固定分类文件齐全，且不存在任务自动创建的额外分类；
- 每个分类文件包含所有必需章节；
- 稳定 ID 在整个资料库内唯一；
- 来源 ID、处理摘要和内部链接有效且无重复；
- 每个输入条目有唯一主分类或进入 `99-unclassified.md`；
- `99-unclassified.md` 条目数量与 `README.md` 一致；
- 相同报告重复输入不会改变分类文件；
- 内容相同但文件名不同的报告不会二次融合；
- 模拟融合失败、校验失败或超时时，正式文件字节不变且 inbox 输入仍存在；
- 首轮删除前，每份历史报告和问题均通过覆盖检查。

日志只输出运行 ID、文件名、摘要、分类结果和计数，不输出报告正文、凭据或其他敏感值。

## 验收标准

- 14 份日期报告已融合并删除，长期目录只包含固定分类文件、`README.md`、`latest.md` 和空的 `inbox/`。
- 问题、趋势、补强点、关键词和来源均有覆盖统计，没有静默丢弃。
- 重复主题只保留一个稳定条目，互补信息已合并，冲突信息已显式标记。
- 定时任务每次先生成 inbox 输入，再依据索引融合；失败不会破坏已有资料或删除输入。
- 固定分类、稳定 ID、幂等摘要、结构和覆盖检查可在无模型参与时重复验证。
- 一份相同的测试报告连续处理两次，第二次不产生内容变化。

## 知识边界

该资料库是本地面试准备材料，不是 Stratum 当前实现的权威说明。仓库代码、测试、ADR 和运行证据仍是项目事实源；市场趋势按来源核验日期理解，自动生成的补强建议必须经代码和运行证据复核后才能进入项目任务或 ADR。
