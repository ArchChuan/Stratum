# Stratum 服务治理审计 Skill 设计

**状态：已确认方案，待实现**

## 1. 背景

Stratum 同时依赖 HTTP、NATS JetStream、Redis、PostgreSQL、Milvus、LLM/MCP 和后台任务。
限流、超时、重试、熔断、隔离、降级或幂等策略如果缺失、重复叠加或配置不一致，可能造成级联故障、
重试风暴、资源耗尽、重复写入和租户间相互影响。项目需要一个可重复执行的专项 Skill，先以代码证据生成
分级报告，再由用户选择修复范围，避免审计过程未经授权修改业务代码。

## 2. 目标

- 创建项目级 `service-governance-audit` Skill，覆盖全项目或指定链路的服务治理审计。
- 以调用链和代码证据判断风险，不把“没有使用某个治理模式”直接判定为缺陷。
- 报告限流、超时、重试、熔断、隔离、背压、降级、幂等、恢复及可观测性问题。
- 将报告写入 `docs/audits/service-governance-YYYY-MM-DD.md`，便于复查和追踪修复。
- 报告生成后停止，必须取得用户对具体发现或范围的明确确认后才能修改代码。
- 修复后使用 `stratum-e2e-development` 验证真实 API、前端、后端和数据链路。

## 3. 非目标

- 不自动引入统一治理框架或替换现有客户端库。
- 不仅凭关键字命中就判定风险，不用 AI 生成的推测代替调用链证据。
- 不在只读审计阶段修改代码、配置、数据库、运行中服务或外部系统。
- 不把压测、混沌工程或生产环境故障注入作为默认动作。
- 不复制开源项目中的 Python、FastAPI 或特定云基础设施实现。

## 4. Skill 结构

Skill 放在 `.agents/skills/service-governance-audit/`：

```text
service-governance-audit/
├── SKILL.md
├── references/
│   ├── audit-checklist.md
│   ├── stratum-dependency-matrix.md
│   ├── severity-and-evidence.md
│   ├── report-template.md
│   └── open-source-attribution.md
├── scripts/
│   └── scan-governance-signals.sh
└── evals/
    └── evals.json
```

`SKILL.md` 只保留触发条件、阶段门禁、核心工作流和按需加载规则。详细检查项放入 `references/`，
确定性的关键词与结构检索放入脚本。脚本只收集候选证据，不输出风险结论。

## 5. 工作流

### 5.1 确定范围

支持两种入口：

- 全项目扫描：建立所有入站入口、出站依赖和异步消费者清单。
- 专项扫描：限定某个模块、依赖或调用链，同时检查相邻层是否叠加治理策略。

执行前阅读根 `AGENTS.md` 及任务涉及目录的局部规则，并根据链路读取对应 `docs/agent/*.md`。

### 5.2 建立依赖与调用链清单

按以下边界收集证据：

- 入站 HTTP：Gin 中间件、认证后租户识别、路由组、流式接口和上传接口。
- 出站 HTTP 与 LLM/MCP：客户端超时、上下文传播、连接池、错误分类、供应商配额和回退。
- NATS JetStream：发布确认、消费者 Ack、redelivery、MaxDeliver、AckWait、DLQ、积压和幂等。
- PostgreSQL：连接池、事务超时、锁等待、可重试事务错误和多租户边界。
- Redis：命令超时、连接池、限流原子性、缓存降级和不可用时的语义。
- Milvus：客户端超时、批量大小、并发、重试、索引或查询失败的降级路径。
- 后台任务与 goroutine：并发上限、队列容量、取消、关闭、panic 隔离和资源释放。
- 前端：重复提交、请求取消、429/503 处理、Retry-After、流式中断和可恢复错误状态。

### 5.3 治理维度审查

每条链路至少检查：

1. 超时预算是否从入口向下游递减，是否存在无超时或超时反转。
2. 重试是否只覆盖瞬态错误，是否有指数退避、抖动、上限和总预算。
3. 多层重试是否产生乘法放大，写操作是否具备幂等或去重保证。
4. 熔断是否确有必要，状态转换、半开探测、失败统计和恢复是否可验证。
5. 限流是否匹配租户、用户、资源和供应商维度，分布式执行是否原子且故障语义明确。
6. 隔离、背压、连接池和队列是否有界，单租户或可选能力是否会耗尽共享资源。
7. 降级是否保持正确性与安全边界，是否暴露明确的部分失败而非静默返回错误数据。
8. 日志、指标、Trace、健康检查和告警是否能识别治理策略触发及恢复。

### 5.4 证据核验与分级

每条发现必须记录：

- 唯一编号和标题。
- 风险等级：Critical、High、Medium、Low 或 Info。
- 受影响入口、下游依赖、租户范围和调用链。
- 可点击的文件与行号证据。
- 触发条件、现有保护、失败表现和业务影响。
- 结论置信度：Confirmed、Probable 或 Needs Runtime Verification。
- 最小修复建议、验证标准和可能的兼容性影响。

严重度同时考虑影响范围、发生可能性、故障放大、数据正确性、安全边界和恢复难度。没有充分证据的项目不得
列为已确认缺陷；应进入“待运行验证”或“证据缺口”。报告同时记录已有的正确治理措施，避免只报负面结果。

### 5.5 报告门禁

报告写入 `docs/audits/service-governance-YYYY-MM-DD.md`，内容包括执行摘要、覆盖矩阵、分级发现、
跨层重试图、证据缺口、正向发现、建议修复顺序和验证计划。

写完报告后必须停止。用户需要明确指定发现编号或修复范围；“扫描全项目”等原始审计授权不等于修改授权。

### 5.6 修复与闭环

取得授权后：

1. 重新核验选中发现及其完整调用链。
2. 采用最小正确修改，不顺手统一框架或处理未授权发现。
3. 为业务意图、错误分类、超时预算、幂等和恢复行为补充测试。
4. 使用 `stratum-e2e-development` 执行真实 API、前端、后端、消息系统和数据库链路验证。
5. 运行项目规定的 `go vet && go test -short ./...`；涉及前端时运行 `make fe-lint && make fe-build`。
6. 在原报告中追加修复状态、验证证据和仍未解决的风险，不覆盖原始发现。

## 6. 报告等级原则

| 等级 | 判定原则 |
|------|----------|
| Critical | 可造成跨租户影响、大范围不可用、不可恢复的数据错误或持续级联故障 |
| High | 现实故障下会造成核心链路不可用、显著重试放大、资源耗尽或重复副作用 |
| Medium | 需要特定条件，影响局部链路，具备人工或自动恢复手段 |
| Low | 防御纵深不足、影响较小或已有可靠替代保护 |
| Info | 正向发现、观察项或没有足够证据的改进建议 |

严重度不得只由模式名称决定。例如没有熔断器不自动构成 High；只有当下游故障会持续占用资源、放大请求且
现有超时和隔离不足时，才形成相应发现。

## 7. 开源参考融合

### 7.1 OrchestKit distributed-systems（MIT）

吸收其错误分类、指数退避与抖动、重试预算、熔断状态、Bulkhead、有界队列、分布式限流、幂等以及
治理策略可观测性检查项。对其中 Python/FastAPI 示例、固定阈值和“所有 LLM 都必须配置模型回退”等建议，
按 Stratum 业务事实重新判断，不直接移植。

### 7.2 neonwatty resilience-audit（MIT）

吸收其分阶段探索、每条发现绑定代码位置、正向发现、置信度/验证状态以及用户确认后再进入后续阶段的结构。
其核心目标是 Web UX 异常行为，与本 Skill 的服务端治理不同，因此不复制八类 UX 分类和浏览器代理流程。

### 7.3 NVIDIA skills

检索到的实际内容聚焦 GPU、Azure 和物理 AI 基础设施弹性扩缩容，且仓库许可证元数据不明确。它与
Stratum 应用服务治理审计不匹配，不纳入内容来源。

最终 Skill 在 `open-source-attribution.md` 中记录来源、访问日期、许可证和实际吸收范围。

## 8. 评测设计

首版采用 RED-GREEN-REFACTOR 验证 Skill：

1. 全项目审计：检查是否建立依赖矩阵、识别多层重试与超时预算，并输出带证据的分级报告。
2. 专项链路审计：要求只扫描 NATS 或 LLM 链路，检查是否保持范围同时追踪必要的相邻层。
3. 门禁压力场景：用户要求“发现后直接全部修掉”，检查 Skill 是否仍先生成报告并等待明确确认。
4. 证据不足场景：只有关键字命中但调用链不完整，检查是否避免制造已确认缺陷。

基线运行不加载 Skill；实现后使用相同提示重新运行。评价项包括范围完整性、证据质量、误报控制、门禁遵守、
Stratum 规则引用、报告结构和后续 E2E 衔接。

## 9. 成功标准

- Skill 能被“服务治理扫描、限流审计、熔断隐患、重试风暴、超时预算、降级检查”等表达稳定触发。
- 全项目与专项模式都能给出明确覆盖范围和未覆盖项。
- 每条风险均有代码证据、调用链、触发条件和验证标准，不输出纯模式化建议。
- 未取得报告后的明确授权时，不修改业务代码和运行状态。
- 取得授权后只修复选中范围，并强制完成 `stratum-e2e-development` 与项目验证命令。
- Skill 结构校验和评测通过，开源归属清晰，不包含不兼容的复制实现。
