# 内置 Stratum 系统助手设计

**状态：Deferred（已完成初步产品设计，暂缓实现）**

## 1. 背景

Stratum 需要为每个租户提供一个平台内置的默认 Agent，作为售后支持、平台搭建和使用指导的统一入口。
该 Agent 暂定名为「Stratum 系统助手」，主要解决三类任务：

1. 诊断当前租户内的系统异常和资源故障。
2. 通过对话配置、创建或更新 Agent、Skill、MCP 和 RAG 知识库等资源。
3. 基于 Stratum 官方资料提供系统使用说明和操作指引。

项目正在另一会话中设计资源自进化调优体系，目前先以 Skill 为切入口。系统助手与自进化体系都可能修改
同一资源，因此本方案暂缓进入技术实现，待 Skill 自进化的版本、评测、自动应用和回滚机制稳定后再继续。

## 2. 目标

- 每个新旧租户都默认拥有且只拥有一个系统助手。
- 系统助手不可删除，核心行为由平台维护，同时允许有限的租户扩展。
- 普通成员可以使用诊断和指引能力，资源变更仍遵守现有租户 RBAC。
- 所有诊断、建议和资源变更都可追踪、可审计，并严格限制在当前租户。
- 后续与资源自进化体系共用版本和变更基础设施，避免形成两套互相覆盖的写入逻辑。

## 3. 已确认的产品设计

### 3.1 系统助手身份

- 固定名称：`Stratum 系统助手`。
- 固定描述：`诊断系统问题、搭建 Agent 能力并指导平台使用`。
- 系统助手属于平台托管的系统 Agent，不等同于新的 Agent 执行算法；执行层仍复用 ReAct。
- Agent 列表应将系统助手置顶并显示「系统内置」标识。
- 名称、系统身份、核心提示词、系统工具和安全规则由平台管理，租户不能删除或覆盖。
- 租户管理员可以选择模型，并追加租户提示词、知识库、Skill 和 MCP。

### 3.2 权限边界

- 普通租户成员可以对话、查询、诊断和查看操作指引。
- 只有租户管理员可以发起和确认资源创建或更新。
- 首版不支持通过系统助手删除任何资源。
- 系统助手不得绕过现有 Agent、Skill、MCP、Knowledge 的管理员权限。
- 系统助手只能访问当前 JWT 对应租户，不能接受模型生成或用户输入的 tenant ID 作为授权依据。

### 3.3 系统异常与故障诊断

首版诊断范围限定为当前租户的应用级状态：

- 租户模型和基础配置是否完整。
- Agent 最近执行记录、执行轨迹、工具调用和脱敏错误。
- Skill 草稿、发布版本、测试和评测状态。
- MCP 服务器连接状态、工具目录和最近连接错误。
- RAG workspace、文档摄取进度和脱敏错误。

首版不读取宿主机、Kubernetes、其他租户或平台全局基础设施日志。诊断结果必须区分已确认事实、推断和
缺失证据；工具不可用时必须暴露失败并降低结论置信度。

### 3.4 资源配置与创建

系统助手计划支持以下受控操作：

- Agent：创建或更新普通 Agent。
- Skill：创建或更新草稿、工具契约、实现草稿和初始测试用例；不自动发布。
- MCP：创建或更新配置；不得在聊天中采集或回显 Token、API Key 等密钥。
- RAG：创建 workspace 或更新允许变更的配置；文档仍通过专用上传界面摄取。

资源变更需要先展示结构化预览和影响范围，再由租户管理员明确确认。不能依赖用户在自然语言中输入
「确定」作为唯一授权凭证，最终确认应由受鉴权的界面动作和后端状态机完成。

### 3.5 使用说明与指引

- 系统助手检索平台维护的只读 Stratum 官方文档索引。
- 回答需附文档标题、版本和可回溯引用，禁止伪造来源。
- 官方产品资料与租户业务知识分离；共享官方文档不得成为跨租户读取业务数据的通道。
- 租户没有配置可用模型时，系统助手仍应可见，并提供明确的模型配置入口和静态引导。

## 4. 与资源自进化体系的关系

### 4.1 已识别的冲突

系统助手和自进化体系可能同时成为资源写入者：

- 系统助手根据用户指令修改资源。
- 自进化引擎根据运行数据和评测结果自动调优同一资源。
- 平台模板升级还可能修改系统助手本身。

如果三条路径独立写入，将产生版本覆盖、旧评测覆盖新配置、循环调优、审计责任不清和无法可靠回滚等问题。

### 4.2 当前原则

- 系统助手是用户主动发起的资源建设和运维入口。
- 自进化引擎是运行数据驱动的持续调优入口。
- 系统助手的核心字段不参与租户级自进化，只能由平台模板升级。
- 系统助手的租户扩展配置是否允许自动调优，待自进化策略稳定后决定。
- 两类入口最终应共用资源版本、冲突检测、审计和回滚机制，不能直接建立彼此独立的写入路径。

曾讨论将统一机制抽象为 `resource_change_proposals`，但该名称、数据模型和是否所有变更都必须经过 proposal
尚未定案。后续应以 Skill 自进化实现验证后的事实为准，避免提前固化错误抽象。

## 5. 暂缓范围

在恢复设计前，不实施以下内容：

- 系统 Agent 的数据库字段和租户自动补齐逻辑。
- 系统助手专属诊断工具和跨 bounded context adapter。
- Agent、Skill、MCP、RAG 的对话式创建或更新。
- 通用资源变更 proposal、审批、自动应用或回滚状态机。
- 系统助手前端入口、系统标签和变更确认卡片。

## 6. 恢复设计的前置条件

Skill 自进化体系至少明确以下事实后，再恢复本方案：

1. Skill 的版本模型以及每次调优生成什么版本或快照。
2. 自动应用的触发条件、允许修改的字段和风险边界。
3. 评测基线、灰度、冷却期、回滚和失败恢复机制。
4. 人工修改与自动调优并发发生时的冲突策略。
5. 审计记录、执行身份和权限校验方式。
6. 可复用于 Agent、MCP 和 RAG 的能力，以及必须保留在 Skill 域内的部分。

## 7. 恢复后的优先工作

1. 对照 Skill 自进化的真实实现，识别可复用的资源版本和变更契约。
2. 决定系统助手的人工确认变更与自动调优是否进入同一个状态机。
3. 明确每种资源的创建、更新、自动应用和禁止操作白名单。
4. 补充系统助手的数据模型、租户初始化、API、前端交互和端到端测试方案。
5. 使用 `stratum-e2e-development` 验证真实 API、前端、后端和数据库链路后再宣布完成。

## 8. 当前非目标

- 不在本方案中重新设计正在推进的 Skill 自进化体系。
- 不允许系统助手自动删除资源、重启服务或修改基础设施。
- 不让 AI 决定权限、审批、重试、状态流转或回滚条件。
- 不在 Skill 自进化机制稳定前创建第二套通用资源变更框架。
